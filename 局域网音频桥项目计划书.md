## 一、项目定位

**一句话定义**：让手机的音频零配置、低延迟、多模式地传到电脑播放。

**目标用户**：大学生、寝室用户、游戏玩家、需要无线麦克风的普通用户。

**核心原则**：用户只管用，网络环境的变化交给软件去处理。

## 二、技术架构

### 全平台架构

| 端 | 语言 | UI框架 | 音频采集 | 音频播放 | 编解码 | 设备发现 |
|---|------|--------|---------|---------|--------|---------|
| **Android** | Kotlin | Jetpack Compose | MediaProjection / AudioRecord | — | Concentus（Java） | NsdManager |
| **Windows** | C# | Avalonia | — | NAudio / VB-CABLE | Concentus | Makaretu.Dns |
| **iOS** ⏳ | Swift | SwiftUI | ReplayKit / AVAudioSession | — | libopus | Bonjour |
| **macOS** ⏳ | Swift | SwiftUI | — | CoreAudio / BlackHole | libopus | Bonjour |

**协议层**：四端共享同一套 UDP + Opus + PacketHeader 协议。

### 数据传输
- 音频端口：UDP 12345，包格式 `[14B PacketHeader][Opus数据]`
- 控制端口：UDP 12347，二进制 PacketHeader 协议

### 音频参数
| 参数 | 值 |
|------|----|
| 采样率 | 48kHz |
| 位深 | 16bit |
| 声道 | 单声道 |
| 帧长 | 20ms（960采样点，1920字节） |
| Opus码率 | 128kbps |
| 编码复杂度 | 10（最高） |
| FEC | 开启，丢包率预期15% |
| VBR模式 | 约束VBR（CVBR） |
| 信号类型 | OPUS_SIGNAL_MUSIC |

## 三、核心功能

### 3.1 四条音频管线（模式）

| 模式 | 采集端（手机） | 输出端（电脑） | 场景 |
|------|--------------|--------------|------|
| **模式1** Speaker | 系统音频 | 扬声器 | 刷抖音不外放 |
| **模式2** Monitor | 系统音频 + 麦克风混音 | 扬声器 | 直播/解說 |
| **模式3** Virtual Mic | 麦克风 | VB-CABLE虚拟麦克风 | 无线麦克 |
| **模式4** Shenanigans | 系统音频 | VB-CABLE虚拟麦克风 | 整活 |

### 3.2 四级链路（场景引导）

| 链路 | 适用场景 | 用户操作 | 状态 |
|------|---------|---------|------|
| 🏠 **在家（WiFi LAN）** | 同路由器，低延迟高带宽 | 零操作（自动发现） | ✅ 已完成 |
| 🌐 **万能（WiFi Direct）** | 无路由器/面对面，性能接近 LAN | 手机扫码，支持配对记忆 | ✅ 已完成 |
| 🔵 **蓝牙** | 近距离应急，延迟较高但省电 | 手动蓝牙配对，手机点“蓝牙直连” | ✅ 已完成 |
| 🔌 **USB** | 有线最稳定，顺便充电 | 插线 + USB调试，手机点“USB直连” | ✅ 已完成 |

> 设计思路：不自动降级。用户通过场景卡片自主选择链路，UI 上清晰标明当前链路和适用环境。

### WiFi Direct P2P 架构设计

```
Windows (P2P Client)                     Android (P2P GO)
────────────────────                     ──────────────────
DeviceWatcher 持续监听                    扫码 → 得到 token
  │                                      │
  ├─ 发现 Android P2P 设备               ├─ createGroup() → GO (192.168.49.1)
  ├─ WiFiDirectDevice.FromIdAsync()      ├─ 监听 UDP 12347
  ├─ 轮询 P2P 适配器 IP                  │
  │                                      │
  ├─ HELLO(token) → 192.168.49.1:12347  ├─ 校验 token → 记录 lastRemoteIp
  ├─ 收到 HELLO_ACK(route)              ├─ 回 HELLO_ACK(route)
  │                                      │
  ├─ AudioEngine ← 12345                ├─ AudioPipeline → Windows P2P IP:12345
  └─ 解码 → 播放                         └─ 采集 → Opus → UDP
```

**核心优势**：建链后数据通路 100% 复用现有 UdpTransport + PacketHeader + AudioEngine，上层协议零改动。

### 架构变化

**重构前：**
```
AudioEngine（398行）：UDP接收 + Opus解码 + JitterBuffer调度 + 看门狗 + 诊断
AudioPipeline（350行）：采集循环 + 混音 + 编码 + 发送 + 看门狗
LinkManager（280行）：LAN/P2P 逻辑杂糅，状态回调+切换+降级混在一起
```

**重构后：**
```
AudioEngine（190行） → 编排：OpusDecodePipeline + PlaybackScheduler + AudioWatchdog
AudioPipeline（93行） → 编排：CaptureLoop + EncodeSender + PcmFrameAssembler
LinkManager（~50行） → 纯路由器，when(linkType) 分发
各链路（WifiLan/WifiDirect/Bluetooth/Usb）→ 独立子目录，互不引用，完全自治
```

### 设计原则

- **不自动降级**：用户通过场景卡片自主选择链路，UI 标明当前链路和适用环境
- **四条链路互相不知道对方存在**，同时只能打开一条
- `LinkManager` 只做 when(linkType) 分发，不包含任何实现代码
