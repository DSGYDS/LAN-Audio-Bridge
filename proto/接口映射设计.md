# 第一期接口（Phase 1）

## 1. ILogger

### 定位
统一日志接口。项目两端当前共 34 个源文件，日志散落在各处：

| 平台 | 当前方式 | 散落文件数 |
|------|---------|-----------|
| Windows | `Console.WriteLine(...)` | 6 个文件：AudioEngine.cs、AudioRouter.cs、HandshakeServer.cs、MdnsPublisher.cs、MainWindow.axaml.cs、BufferStats |
| Android | `Log.i/e/d(...)` | 5 个文件：AudioPipeline.kt、MicrophoneCapturer.kt、SystemAudioCapturer.kt、HandshakeManager.kt、ReconnectionManager.kt |

### 接口定义

```csharp
// —— C# ——
public enum LogLevel { Debug, Info, Warn, Error }

public interface ILogger
{
    void Debug(string tag, string msg);
    void Info(string tag, string msg);
    void Warn(string tag, string msg);
    void Error(string tag, string msg);
    void Error(string tag, string msg, Exception? ex = null);
}

// 静态门面（避免构造函数传递）
public static class Log
{
    private static ILogger _impl = new ConsoleLogger();
    public static void SetImpl(ILogger impl) => _impl = impl;
    public static void D(string tag, string msg) => _impl.Debug(tag, msg);
    public static void I(string tag, string msg) => _impl.Info(tag, msg);
    public static void W(string tag, string msg) => _impl.Warn(tag, msg);
    public static void E(string tag, string msg) => _impl.Error(tag, msg);
    public static void E(string tag, string msg, Exception ex) => _impl.Error(tag, msg, ex);
}
```

```kotlin
// —— Kotlin ——
```

## 2. IPacketProtocol

### 定位
统一协议编解码接口。当前已有完整实现：

| 文件 | 行数 | 功能 |
|------|------|------|
| `PacketHeader.cs` | ~120 行 | 14B 二进制 header 编解码 |
| `PacketHeader.kt` | ~120 行 | 同上 |
| `ProtocolTypes.cs` | 14 行 | PacketType 枚举 |
| `ProtocolTypes.kt` | 20 行 | PacketType 枚举 |

现有 API 和接口的映射：

```
PacketHeader.Encode(type, seq, payload)  →  IPacketProtocol.Encode(in Packet)
PacketHeader.TryDecode(data)              →  IPacketProtocol.Decode(ReadOnlySpan<byte>)
```

`PacketHeader` 已经是纯编解码工具类，**加一个实现层即可，不修改原有**逻辑。

### 接口定义

```csharp
// —— C# ——
public struct Packet
{
    public PacketType Type;          // 沿用现有 ProtocolTypes.cs 的枚举
    public ushort Sequence;          // 帧序号
    public byte[] Payload;           // 载荷
}

public interface IPacketProtocol
{
    byte[] Encode(in Packet packet);
    Packet? Decode(ReadOnlySpan<byte> data);
}
```

```kotlin
// —— Kotlin ——
data class Packet(
    val type: PacketType,
    val sequence: UShort,
    val payload: ByteArray
)

interface IPacketProtocol {
    fun encode(packet: Packet): ByteArray
    fun decode(data: ByteArray): Packet?
}
```

### 映射表

| 现有方法 | 接口方法 | 适配策略 |
|---------|---------|---------|
| `PacketHeader.Encode(type, seq, payload)` → `byte[]` | `IPacketProtocol.Encode(Packet)` | **包裹**：`PacketHeaderAdapter.Encode(p)` → `PacketHeader.Encode(p.Type, p.Sequence, p.Payload)` |
| `PacketHeader.TryDecode(data)` → `PacketHeaderInfo?` | `IPacketProtocol.Decode(data)` → `Packet?` | **包裹**：`PacketHeaderAdapter.Decode(data)` → 转 `PacketHeaderInfo` 为 `Packet` |

### 设计决策

1. **不改 PacketHeader**。`PacketHeader` 保持纯工具类，新增 `PacketHeaderAdapter : IPacketProtocol` 实现接口。
2. **返回 `Packet?` 而非异常**。解析失败返回 null，与现有 `TryDecode` 行为一致。
3. **不与序列化框架耦合**。`Encode` 不处理 Version（写死在 PacketHeader 内部），业务层不感知协议版本。

### 新增文件

Windows: `Core/Interfaces/IPacketProtocol.cs`, `Core/Models/Packet.cs`, `Core/Adapters/PacketHeaderAdapter.cs`
Android: `core/interfaces/IPacketProtocol.kt`, `core/models/Packet.kt`, `core/adapters/PacketHeaderAdapter.kt`

## 3. ITransport

### 定位
统一网络传输接口。这是整个传化层的核心接口，四级链路切换的基础。

当前项目中 UDP 传输的使用者：

| 文件 | 角色 | 端口 | 收/发 |
|------|------|------|-------|
| `AudioEngine.cs` | Windows 音频接收 | 12345 | 只收（UdpClient.Receive） |
| `HandshakeServer.cs` | Windows 握手 | 12347 | 收发 |
| `UdpSender.kt` | Android 音频发送 | 12345 | 只发（DatagramSocket） |
| `HandshakeManager.kt` | Android 握手 | 12347 | 收发 |

### 接口定义

```csharp
// —— C# ——
public enum TransportType { Udp, WifiDirect, Usb, Hotspot, Relay }

public interface ITransport
{
    /// <summary>连接到远端。参数在构造时传入（host + port）。</summary>
    Task ConnectAsync(CancellationToken ct = default);

    /// <summary>断开连接，释放端口。</summary>
    Task DisconnectAsync();

    /// <summary>发送数据。</summary>
    Task SendAsync(ReadOnlyMemory<byte> data, CancellationToken ct = default);

    /// <summary>收到数据包时触发。</summary>
    event Action<ReadOnlyMemory<byte>>? PacketReceived;

    /// <summary>当前是否已连接。</summary>
    bool IsConnected { get; }

    /// <summary>传输类型（Udp / WifiDirect / ...）。</summary>
    TransportType Type { get; }
}
```

```kotlin
// —— Kotlin ——
interface ITransport {
    suspend fun connect()
    suspend fun disconnect()
    suspend fun send(data: ByteArray)

    /** 收到数据包的回调。单订阅者版本，需要多订阅者时升级 SharedFlow。 */
    var onPacketReceived: ((ByteArray) -> Unit)?

    val isConnected: Boolean
    val type: TransportType
}
```

### 映射表

#### 1) Windows AudioEngine.cs（UdpClient 接收端）

```csharp
// 现有代码（简化）
private UdpClient _udp = new(12345);
// 在 DecodeLoop 中:
byte[] data = _udp.Receive(ref ep);

// 映射到 ITransport 后
private ITransport _transport;  // UdpTransport(host: any, port: 12345, isServer: true)
_transport.PacketReceived += OnPacketReceived;
```

| AudioEngine.cs 现有成员 | 映射 |
|------------------------|------|
| `UdpClient _udp` | → `UdpTransport(any, 12345)` |
| `_udp.Receive(ref ep)` | → `PacketReceived` 事件 |
| `_udp.Close()` / `Dispose()` | → `DisconnectAsync()` |

#### 2) Windows HandshakeServer.cs（UdpClient 握手端）

| HandshakeServer.cs 现有成员 | 映射 |
|---------------------------|------|
| `UdpClient _sock` (port 12347) | → `UdpTransport(any, 12347)` |
| `sock.Receive(ref ep)` | → `PacketReceived` 事件 |
| HELLO 回复（`_sock.Send(...)`） | → `SendAsync(...)` |

#### 3) Android UdpSender.kt（DatagramSocket 发送端）

```kotlin
// 现有代码
class UdpSender {
    private var sock: DatagramSocket? = null
    fun prepare(host: String, port: Int) { ... }
    fun send(data: ByteArray) { ... }
    fun release() { ... }
}

// 映射到 ITransport 后
// UdpTransport 实现 ITransport，包裹 DatagramSocket
// AudioPipeline 改持 ITransport 引用
```

| UdpSender.kt 现有成员 | 映射 |
|----------------------|------|
| `prepare(host, port)` | → `UdpTransport(host, port)` 构造，`connect()` |
| `send(data)` | → `send(data)` |
| `release()` | → `disconnect()` |

#### 4) Android HandshakeManager.kt（握手端）

| 现有 UDP 操作 | 映射 |
|-------------|------|
| `DatagramSocket()` (port 12347) | → `UdpTransport(any, 12347)` |
| `socket.send(packet)` | → `SendAsync(...)` |
| `socket.receive(packet)` | → `PacketReceived` 事件 |

### 设计决策

1. **每端口一个 Transport 实例**。握手（12347）和音频（12345）各自独立 `UdpTransport`，各自维护连接状态，不共享 UdpClient。
2. **构造时注入地址和端口**。`ITransport` 的 `ConnectAsync()` 无参（参数在构造时确定），与简化后的 ITransport 接口一致。
3. **音频 Transport 无自动重连**。重连逻辑由 `ReconnectionManager` 控制，Transport 层只做"连接/断连/收发"。
4. **Kotlin 用回调而非 Flow**。单订阅者场景用 `var onPacketReceived: ((ByteArray) -> Unit)?` 最简单。如需多订阅者，Adapter 内部做 fan-out。

### 新增文件

Windows:
- `Core/Interfaces/ITransport.cs`
- `Core/Enums/TransportType.cs`
- `Core/Adapters/UdpTransport.cs`

Android:
- `core/interfaces/ITransport.kt`
- `core/enums/TransportType.kt`
- `core/adapters/UdpTransport.kt`

### 适配后涉及改动的文件

| 文件 | 改动量 |
|------|--------|
| `AudioEngine.cs` | 中（UdpClient → ITransport，接收循环改为事件） |
| `HandshakeServer.cs` | 中（UdpClient → ITransport） |
| `UdpSender.kt` | 大（类改为实现或委托给 UdpTransport） |
| `HandshakeManager.kt` | 中（DatagramSocket → ITransport） |
| `AudioPipeline.kt` | 小（UdpSender 引用改为 ITransport 引用） |

# 第二期接口（Phase 5）

## 4. IAudioCapturer

### 定位
统一音频采集接口。**纯 PCM 层**——采集原始 PCM16LE 帧，不涉及 Opus 编解码。

当前项目中采集功能的分布：

| 文件 | 采集源 | 输出 |
|------|--------|------|
| `MicrophoneCapturer.kt` | AudioRecord（麦克风） | PCM16LE byte[] |
| `SystemAudioCapturer.kt` | MediaProjection（系统音频） | PCM16LE byte[] |
| Windows 端 | 无采集功能（纯接收端） | N/A |

### 接口定义

```csharp
// —— C# ——
public enum CapturerType { Microphone, SystemAudio, Mixed }

public interface IAudioCapturer
{
    /// <summary>准备采集器（分配资源）。</summary>
    bool Prepare(AudioConfig config);

    /// <summary>开始采集。</summary>
    bool Start();

    /// <summary>读取一帧 PCM16LE，返回实际读取的字节数（bytesRead）。</summary>
    int ReadFrame(byte[] buffer, int offset, int count);

    /// <summary>停止采集。</summary>
    void Stop();

    /// <summary>释放所有资源。</summary>
    void Release();

    /// <summary>采集源类型。</summary>
    CapturerType SourceType { get; }
}
```

```kotlin
// —— Kotlin ——
interface IAudioCapturer {
    fun prepare(config: AudioConfig): Boolean
    fun start(): Boolean
    fun readFrame(buffer: ByteArray, offset: Int, count: Int): Int
    fun stop()
    fun release()
    val sourceType: CapturerType
}
```

### 映射表

| 现有方法 | 接口方法 | 适配策略 |
|---------|---------|---------|
| `MicrophoneCapturer(AudioConfig)` | → 构造时传入 config，`prepare()` 中初始化 | 包裹 |
| `MicrophoneCapturer.start()` | → `start()` | 直接映射 |
| `MicrophoneCapturer.readFrame(buf, offset)` | → `readFrame(buf, offset, count)` | 直接映射 |
| `MicrophoneCapturer.stop()` | → `stop()` | 直接映射 |
| `SystemAudioCapturer(AudioConfig)` | → 同上 | 同上 |
| `SystemAudioCapturer.start(projection)` | → `start()`（proj 通过构造或 prepare 传入） | **缝隙**：当前需要传 `MediaProjection` 参数 |
| Hpf 内部类（MicrophoneCapturer 的高通滤波器） | → 保持内部，capturer 自带 | 不暴露到接口 |

### 缝隙与决策

1. **`SystemAudioCapturer.start(MediaProjection)`**：`MediaProjection` 是 Android 特有类型，接口不能接收它。解决方案：`MediaProjection` 通过构造或 `prepare(config, projection)` 传入 adapter，不污染接口定义。
2. **Windows 端**：当前不采集音频。Phase 5 新增 `StubCapturer`，所有方法抛出 `NotSupportedException`。以后 iOS/macOS 加入时各自实现。
3. **`Mixed` 类型**：当前 `AudioMixer.mix()` 由 `AudioPipeline` 调用，非 capturer 职责。capturer 接口保持单一采集源。

### 新增文件

Windows: `Core/Interfaces/IAudioCapturer.cs`, `Core/Enums/CapturerType.cs`
Android: `core/interfaces/IAudioCapturer.kt`, `core/enums/CapturerType.kt`

### Adapter

| 平台 | Adapter | 说明 |
|------|---------|------|
| Android | `MicCapturerAdapter` | 包裹 `MicrophoneCapturer` |
| Android | `SystemAudioCapturerAdapter` | 包裹 `SystemAudioCapturer`，projection 通过构造传入 |
| Windows | `StubCapturer` | 占位，throw NotSupported |

## 5. IAudioRenderer

### 定位
统一音频渲染接口。提供 PCM16LE 到扬声器/虚拟麦克风的输出。

当前项目中渲染功能的分布：

| 文件 | 渲染目标 | 输入 |
|------|---------|------|
| `AudioRouter.cs` | WaveOut（扬声器）+ WasapiOut（CABLE 虚拟麦克风） | PCM16LE byte[] |
| Android 端 | 无渲染功能（纯发送端） | N/A |

### 接口定义

```csharp
// —— C# ——
public interface IAudioRenderer
{
    bool Prepare(AudioConfig config);
    void Play();
    void Stop();
    void SetVolume(float volume);   // 0.0 ~ 1.0
    void Mute(bool muted);
    void FeedPcm(byte[] data, int offset, int count);
    void Release();
}
```

```kotlin
// —— Kotlin ——
interface IAudioRenderer {
    fun prepare(config: AudioConfig): Boolean
    fun play()
    fun stop()
    fun setVolume(volume: Float)
    fun mute(muted: Boolean)
    fun feedPcm(data: ByteArray, offset: Int, count: Int)
    fun release()
}
```

### 映射表

| AudioRouter.cs 现有成员 | 接口方法 |
|------------------------|---------|
| `SetMode(SpeakerOnly/MicOnly/...)` | → **不在接口层**。路由模式是业务逻辑，`AudioPipeline`/`ConnectionManager` 持有多个 `IAudioRenderer` 实例来实现路由切换 |
| `OnAudioFrame(byte[] pcm)` 内部分发到 `_speakerBuf`/`_micBuf` | → `feedPcm(data, offset, count)` |
| `_speakerOut.Play()` | → `Play()` |
| `_speakerOut.Stop()` | → `Stop()` |
| `Mute()` 无现有实现 | → `Mute(bool)` 新增 |

### 设计决策

1. **IAudioRenderer 不包含路由逻辑**。当前 `AudioRouter.RouteMode`（SpeakerOnly / MicOnly / Both / MicOnlySys）是业务组合逻辑，不是渲染能力。Phase 6 重构后，`AudioPipeline` 会持有 2 个 IAudioRenderer 实例（扬声器 + 虚拟麦克风），由 Pipeline 决定哪个喂数据。
2. **Windows 端拆分**：当前 `AudioRouter` 同时管理扬声器和虚拟麦克风。适配时需要拆成两个 IAudioRenderer 实例。
3. **Android 端**：当前不渲染音频。新增 `StubRenderer`。

### 新增文件

Windows: `Core/Interfaces/IAudioRenderer.cs`
Android: `core/interfaces/IAudioRenderer.kt`

### Adapter

| 平台 | Adapter | 说明 |
|------|---------|------|
| Windows | `SpeakerRenderer` | 包裹 WaveOutEvent + BufferedWaveProvider |
| Windows | `CableRenderer` | 包裹 WasapiOut + BufferedWaveProvider（CABLE Input 端点） |
| Android | `StubRenderer` | 占位，throw NotSupported |

## 6. IDiscovery

### 定位
统一设备发现接口。当前两端各有一套发现机制：

| 平台 | 技术 | 文件 |
|------|------|------|
| Windows | Makaretu.Dns (mDNS) | `MdnsPublisher.cs` |
| Android | NsdManager | `LanAudioDiscovery.kt` |

### 接口定义

```csharp
// —— C# ——
public struct DeviceInfo
{
    public string Name;
    public string Ip;
    public int Port;
    public TransportType Transport;
}

public interface IDiscovery
{
    void Start();
    void Stop();
    event Action<DeviceInfo>? OnDeviceFound;
    event Action<DeviceInfo>? OnDeviceLost;
}
```

```kotlin
// —— Kotlin ——
data class DeviceInfo(
    val name: String,
    val ip: String,
    val port: Int,
    val transport: TransportType
)

interface IDiscovery {
    fun start()
    fun stop()
    var onDeviceFound: ((DeviceInfo) -> Unit)?
    var onDeviceLost: ((DeviceInfo) -> Unit)?
}
```

### 映射表

| 现有方法 | 接口方法 | 适配策略 |
|---------|---------|---------|
| `MdnsPublisher.Start()` | → `start()` | 包裹 |
| `MdnsPublisher.Stop()` | → `stop()` | 包裹 |
| `MdnsPublisher` 没有设备事件（只管发布，不管发现） | → **缝隙**：Windows 端当前是被发现方，不是发现方 | 先只实现 Advertise 部分 |
| `LanAudioDiscovery.discoverDevices()` | → `start()` + `onDeviceFound` | 包裹 |
| `NsdManager.DiscoveryListener.onServiceFound` | → `onDeviceFound` | 桥接 |
| `NsdManager.DiscoveryListener.onServiceLost` | → `onDeviceLost` | 桥接 |

### 缝隙与决策

1. **Windows 端是被发现方**（发布 mDNS 服务），Android 是发现方（通过 NsdManager 扫描）。`IDiscovery` 接口设计为"发现方"视角。Windows 端的发布功能保留在 `MdnsPublisher` 中（或作为 `IDiscovery` 的一个"发布+发现"实现），但接口只承诺发现能力。
2. **Phase 5 先只改 Android 端**让 `LanAudioDiscovery` 实现 `IDiscovery`。Windows 端 `MdnsPublisher` 保持独立，等以后 Windows 也需要发现设备时再适配。

### 新增文件

Windows: `Core/Interfaces/IDiscovery.cs`, `Core/Models/DeviceInfo.cs`
Android: `core/interfaces/IDiscovery.kt`, `core/models/DeviceInfo.kt`

| 现有代码 | 接口方法 | 适配策略 |
|---------|---------|---------|
| `ReconnectionManager` 的 `ConnectivityManager.NetworkCallback` | → `onNetworkChanged` | **提取**：将网络监听逻辑从 ReconnectionManager 拆分到 `INetworkMonitor` |
| `NetworkCallback.onAvailable` / `onLost` | → `onNetworkChanged` | 桥接 |
| `NetworkCallback.onCapabilitiesChanged` | → 更新 `Quality` / `TransportType` | 桥接 |

### 设计决策

1. **从 ReconnectionManager 拆分**。当前 `ReconnectionManager` 同时做"网络监听"和"断线重连"两件事。重构后：
   - `INetworkMonitor` → 只做网络状态监听
   - `ReconnectionManager` → 基于 `INetworkMonitor` 事件触发重连
2. **Quality 评估策略**。Phase 5 先简单实现：
   - `Disconnected`: 网络不可用
   - `Poor`: 仅蜂窝数据（WiFi Direct 场景下 WiFi 断开）
   - `Good`: WiFi 已连接且互联网可达
   - `Excellent`: 局域网内低延迟
3. **Windows 端**：Phase 5 新增 `WinNetworkMonitor`，基于 `System.Net.NetworkInformation.NetworkChange`。

### 新增文件

Windows: `Core/Interfaces/INetworkMonitor.cs`, `Core/Enums/NetworkQuality.cs`, `Core/Models/NetworkInfo.cs`
Android: `core/interfaces/INetworkMonitor.kt`, `core/enums/NetworkQuality.kt`, `core/models/NetworkInfo.kt`

# 统一状态机（Phase 3）

### 定位
统一连接状态管理。当前两端各有一套 `ConnectionStateManager`：

| 文件 | 状态数 | 功能 |
|------|--------|------|
| `ConnectionStateManager.cs` | 8 个状态 | enum + isValidTransition + reason + OnStateChanged 事件 |
| `ConnectionStateManager.kt` | 8 个状态 | 同上 |
| `ConnectionState.cs` | 8 个 enum 值 | Disconnected / Searching / Found / Connecting / Connected / Streaming / Recovering / Failed |
| `ConnectionState.kt` | 同上 | 同上 |

### 新状态机定义

```
Idle ──→ Searching ──→ Found ──→ Connecting ──→ Connected ──→ Streaming
  ↑          ↑                                          ↓            ↓
  │          └───────────────────────────────────── Reconnecting ───┘
  │                                                       ↓
  └───────────────── Disconnected ←──────────────────────┘

  Error —— 可从任意状态进入（除 Disconnected 外）
  Disconnected —— 可从任意状态进入
```

### 状态迁移表

| 当前状态 | 可进入状态 | 说明 |
|---------|-----------|------|
| `Idle` | `Searching`, `Disconnected`, `Error` | 初始状态 |
| `Searching` | `Found`, `Disconnected`, `Error` | 扫描中 |
| `Found` | `Connecting`, `Disconnected`, `Error` | 发现设备 |
| `Connecting` | `Connected`, `Disconnected`, `Error` | 握手中 |
| `Connected` | `Streaming`, `Disconnected`, `Error` | 握手成功 |
| `Streaming` | `Reconnecting`, `Connected`, `Disconnected`, `Error` | 音频传输中 |
| `Reconnecting` | `Streaming`, `Searching`, `Disconnected`, `Error` | 断线恢复中 |
| `Disconnected` | `Idle`, `Searching`, `Error` | 已断开（可重新开始） |
| `Error` | `Idle`, `Disconnected` | 故障（需手动恢复） |

### 映射表

#### ConnectionStateManager.cs（Windows）

| 现有 enum 值 | 新 enum 值 | 备注 |
|-------------|-----------|------|
| `Disconnected` | `Idle` | 拆分为两个状态：Idle=初始、Disconnected=主动断开后 |
| `Searching` | `Searching` | 不变 |
| `Found` | `Found` | 不变 |
| `Connecting` | `Connecting` | 不变 |
| `Connected` | `Connected` | 不变 |
| `Streaming` | `Streaming` | 不变 |
| `Recovering` | `Reconnecting` | 改名以对齐四级链路术语 |
| `Failed` | `Error` | 改名 |

#### 改动清单

| 文件 | 改动 |
|------|------|
| `ConnectionState.cs` | 加 `Idle`，`Recovering` → `Reconnecting`，`Failed` → `Error` |
| `ConnectionStateManager.cs` | 更新 `isValidTransition` 表 |

# PlatformFactory（Phase 3）

### 定位
统一工厂，根据平台 + 配置创建所有 Adapter。

### 接口定义

```csharp
// —— C# ——
public static class PlatformFactory
{
    // 第一期（Phase 3 实现）
    public static ILogger CreateLogger();
    public static ITransport CreateTransport(TransportType type, string? host, int port);
    public static IPacketProtocol CreateProtocol();

    // 第二期（Phase 6 实现）
    public static IAudioCapturer CreateCapturer(CapturerType type, ...);
    public static IAudioRenderer CreateRenderer(RendererType type, ...);
    public static IDiscovery CreateDiscovery();
    public static INetworkMonitor CreateNetworkMonitor();
}
```

```kotlin
// —— Kotlin ——
object PlatformFactory {
    fun createLogger(): ILogger = LogcatLogger()
    fun createTransport(type: TransportType, host: String?, port: Int): ITransport
    fun createProtocol(): IPacketProtocol = PacketHeaderAdapter()
    // 第二期...
}
```

### 映射表

| 调用方 | 工厂方法 | 返回实例 |
|--------|---------|---------|
| `MainWindow.axaml.cs` | `CreateLogger()` | `ConsoleLogger` |
| `MainWindow.axaml.cs` | `CreateTransport(Udp, null, 12345)` | `UdpTransport(any, 12345)` — 音频接收 |
| `MainWindow.axaml.cs` | `CreateTransport(Udp, null, 12347)` | `UdpTransport(any, 12347)` — 握手 |
| `AudioEngine.cs` | 内部持有 `ITransport`，通过构造注入 | 由 MainWindow 创建传入 |
| `HandshakeServer.cs` | 同上 | 同上 |
| `AudioPipeline.kt` | `CreateTransport(Udp, host, 12345)` | `UdpTransport(host, 12345)` — 音频发送 |
| `HandshakeManager.kt` | `CreateTransport(Udp, host, 12347)` | `UdpTransport(host, 12347)` — 握手 |

### Key Decision

**Factory 不在运行期根据类型动态创建**，而是在启动时由顶层代码（MainWindow/MainActivity）创建并注入下层模块。这样保持依赖清晰，不引入 Service Locator 反模式。
