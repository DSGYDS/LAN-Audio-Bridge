### UdpTransport 设计要点

```csharp
public sealed class UdpTransport : ITransport, IDisposable
{
    private readonly UdpClient _client;
    private readonly string? _remoteHost;
    private readonly int _remotePort;
    private CancellationTokenSource? _cts;
    private Task? _receiveLoop;

    public UdpTransport(string? remoteHost, int localPort, int? remotePort = null)
    {
        _client = new UdpClient(localPort);
        _remoteHost = remoteHost;
        _remotePort = remotePort ?? localPort;
    }

    public async Task ConnectAsync(CancellationToken ct)
    {
        if (_remoteHost != null)
            _client.Connect(_remoteHost, _remotePort);
        _cts = new CancellationTokenSource();
        _receiveLoop = ReceiveLoopAsync(_cts.Token);
    }

    public async Task SendAsync(ReadOnlyMemory<byte> data, CancellationToken ct)
    {
        await _client.SendAsync(data.ToArray(), data.Length);
    }

    private async Task ReceiveLoopAsync(CancellationToken ct)
    {
        while (!ct.IsCancellationRequested)
        {
            var result = await _client.ReceiveAsync(ct);
            PacketReceived?.Invoke(result.Buffer);
        }
    }

    public event Action<ReadOnlyMemory<byte>>? PacketReceived;
    public bool IsConnected => _cts != null && !_cts.IsCancellationRequested;
    public TransportType Type => TransportType.Udp;

    public async Task DisconnectAsync()
    {
        _cts?.Cancel();
        if (_receiveLoop != null) await _receiveLoop;
        _client.Close();
    }
}
```

### 验证标准

1. **单元测试级**：
   - `UdpTransport.ConnectAsync()` 绑定端口成功
   - `SendAsync` + `PacketReceived` 本地环回收发正确
   - `DisconnectAsync()` 后端口释放，可重新绑定

2. **集成测试级**：
   - HELLO 握手：Android 发 HELLO → Windows 收 → 回 HELLO_ACK，两端状态正确

- **回退方案**：修改前为每个文件创建备份。如果某端替换后出问题，回退到备份，改用"共存过渡方案"（ITransport 和原生 API 共存，逐步迁移）。

```
├── SetMode(SpeakerOnly/MicOnly/Both/MicOnlySys)
└── OnAudioFrame(pcm) → 根据 mode 决定喂给谁
```

**After：**
```
AudioRouter（降级为路由决策器）
├── 持有 IAudioRenderer _speaker（从 PlatformFactory 注入）
├── 持有 IAudioRenderer _mic（从 PlatformFactory 注入）
├── SetMode(SpeakerOnly/MicOnly/Both/MicOnlySys) → 切换内部路由表
└── OnAudioFrame(pcm) → 根据路由表 feedPcm 到一个或两个 renderer
```

这样 AudioRouter 不再直接创建 NAudio 对象，所有平台 API 操作封装在 `SpeakerRenderer` / `CableRenderer` 中。

### 验证标准

- 所有 P1-P4 的验证项**全部重新跑一遍**
- 新增验证：
  - 音频质量无退化（延迟 ≈ 200ms ± 50ms，无爆音卡顿）
  - 采集源切换（MIC ↔ SYSTEM ↔ MIX）无卡顿
  - 网络状态变化能触发 `INetworkMonitor.OnNetworkChanged`
  - 前置 Service（StreamingService / MediaProjectionService）行为不变

### 风险 & 回退

- **高风险**。AudioRouter 拆分影响 Windows 端核心音频路径，Android 端 Capturer 适配影响采集路径。
- **分批策略**：
  1. 先适配 Windows 端 IAudioRenderer（SpeakerRenderer + CableRenderer），验证扬声器和 CABLE 输出正常
  2. 再适配 Android 端 IAudioCapturer（MicCapturerAdapter + SystemAudioCapturerAdapter），验证采集正常
  3. 然后适配 IDiscovery（Android 端），验证设备发现正常
  4. 最后适配 INetworkMonitor，验证网络状态监听正常
- **回退方案**：如果 AudioRouter 拆分后音频路径出问题，`AudioRouter` 保留原有实现作为 fallback，新接口作为可选路径共存。
