## 🛠️ WiFi LAN 直连优化进度（2026-07-23）

### ✅ 已完成

#### 1. 冷启动一帧即停修复
**根因**：JitterBuffer 的 `Pull()` 在 underrun 时无条件执行 `_nextPullSeq++`，冷启动时 HAL 帧到达间隔 ~100ms（正常 20ms），`_nextPullSeq` 飞速前进，后续到达的帧全部被 `Push()` 判定为“迟到包”（`SeqBefore(seq, _nextPullSeq)` → backward < 100）丢弃，形成恶性循环。

**修复**（`JitterBuffer.cs`）：
- underrun 不再立即推进 `_nextPullSeq`，改为连续 N 次后才跳过
- 冷启动阶段（前 100 次拉取 ≈ 2s）：阈值 = 25（500ms 容忍）
- 正常阶段：阈值 = 5（100ms 容忍）
- 成功拉取后重置计数器

#### 2. HELLO 会话重置
**根因**：Android 每次 HELLO 后重启推流（seq 从 0 开始），但 Windows 端 JitterBuffer 的 `_nextPullSeq` 仍停留在上一会话（如 97），新会话 seq=0 被判定为迟到包丢弃。

**修复**（`AudioEngine.cs` + `MainWindow.axaml.cs`）：
- 新增 `AudioEngine.ResetSession()` 方法，重置 JitterBuffer + seq 追踪 + 看门狗状态
- `OnHandshakeRoute` 收到 HELLO 时调用 `ResetSession()`

#### 3. 冷启动淡入
**目的**：掩盖 HAL 冷启动初始 underrun 静音间隙。

**实现**（`AudioEngine.cs`）：
- 每次 `ResetSession()` 后，前 100 帧（≈2s）音量从 0 线性渐变到 1
- 在 `PlaybackTick()` 中对 PCM 数据乘以增益系数

#### 4. Android Opus 编码器预热
**根因**：Opus 编码器首次调用触发 JIT 编译，阻塞采集线程 ~2.3 秒。

**修复**（`AudioEncoder.kt`）：
- `prepare()` 中新增一帧静音预编码，提前触发 JNI JIT 编译

#### 5. Android HAL 预热
**修复**（`MicrophoneCapturer.kt` / `SystemAudioCapturer.kt`）：
- 新增 `warmup()` 方法，采集线程启动后丢弃前 3 帧（≤50ms）
- 新增 `restart()` 方法，看门狗触发时重建 AudioRecord

#### 6. P0 管线被杀自动恢复（已实现，待部署验证）
- 帧时序看门狗：连续 15 次 null（≈300ms）触发 capturer restart，最多 5 次
- Partial WakeLock（2h 上限）+ AudioFocus（AUDIOFOCUS_GAIN）
- AndroidManifest 添加 WAKE_LOCK 权限

#### 7. 诊断日志系统
- `ConsoleLogger` 增加文件日志输出（`debug.log`）
- `AudioEngine` 每 5s 输出帧统计（recv/decOk/pullOk/jbCount/prefilled/speakerReady）
- `JitterBuffer` 输出 Push 丢弃原因（late/dup/overflow）和 Pull GAP 检测

### 🔲 待收尾优化（UI 完成后、打包发布前必须完成）

#### 管线2（Monitor）异常调用虚拟麦克风 ✅ 已修复
**现象**：管线2（系统音频+麦克风→扬声器）模式下会异常触发 CABLE Input（虚拟麦克风）设备。
**根因**：route 1 映射到 `RouteMode.Both`（同时输出到扬声器+CABLE），但管线2 的混音在 Android 端完成，Windows 端只需播放到扬声器。
**修复**：`WifiLanLink.cs` route 1 映射改为 `RouteMode.SpeakerOnly`。
**状态**：✅ 已修复（P5.5.5 验收后）

#### 冷启动初始卡顿深度优化
**现状**：冷启动最初 ~1-2 秒仍有轻微卡顿（已通过淡入掩盖，用户可接受）。

**根因**：Android Audio HAL 冷态导致前 1-2 秒帧到达间隔不稳定（~50-150ms 而非正常 20ms），Windows 端 JitterBuffer 在初始阶段出现 underrun（缓冲耗尽）。

**后期优化方向**：
1. **Android 端增强 warmup**：增加预热帧数（当前 3 帧 → 10+ 帧），确保 HAL 完全就绪后再开始正式采集
2. **Android 端 Opus 编码器深度预热**：在 `prepare()` 中预编码多帧（触发完整 JIT 编译路径）
3. **Windows 端自适应 preFill**：冷启动时动态提高 preFillCount（如检测到帧到达间隔 > 50ms 时提高至 15-20），稳定后降回 5
4. **Windows 端 PLC（丢包隐藏）**：underrun 时用 Opus PLC 生成 comfort noise 而非纯静音
5. **Android 端采集线程提权**：`Process.setThreadPriority(THREAD_PRIORITY_URGENT_AUDIO)` 减少初始帧调度抖动

#### 推流中切出/冷启动其他 App 导致卡顿
**现状**：推流中切桌面或冷启动其他应用时，音频出现短暂卡顿（进程不再被杀，但帧发送节奏被打乱）。

**根因**：
1. **GC 暂停**：`readFrame()` 每帧分配新 `ByteArray(1920)`，导致频繁 GC（每 2-3s 一次），单次 GC 暂停可达 **185ms**
2. **CPU 抢占**：其他 App 冷启动占用大量 CPU，采集线程调度延迟，帧突发到达 Windows 端

**修复方案**：
1. **缓冲区复用**：`readFrame()` 改为复用预分配的 ByteArray，消除每帧 GC 压力
2. **采集线程提权**：`Process.setThreadPriority(THREAD_PRIORITY_URGENT_AUDIO)` 确保 CPU 抢占时采集线程优先调度
3. **Windows 端 JitterBuffer 自适应**：检测到帧突发时动态加深缓冲，平滑突发输出

## 🏗️ WiFi Direct P2P 链路开发（2026-07-23 启动）

### 架构定稿

- **Android 做 GO**（Group Owner），固定 IP `192.168.49.1`
- **Windows 做客户端**，通过 DeviceWatcher 发现并连接 Android P2P 设备
- **QR 码**：`LABRIDGE://version=1&transport=wifidirect&device=XXX&token=XXX`（仅含设备名 + 随机 8 位 hex token，不含 IP/SSID/密码）
- **握手方向**：Windows 主动向 Android 固定 IP 发 HELLO(token)，Android 校验后回 HELLO_ACK(route)
- **数据通路**：建链后 100% 复用现有 UdpTransport + PacketHeader + AudioEngine，上层协议零改动
